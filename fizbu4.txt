1. Design Principles
Fizbo's users are anxious. Selling a property is one of the highest-stakes financial decisions in a person's life. The UX must project calm confidence, reduce cognitive load, and make every step feel achievable.
 
Design north star:  "Professional guidance without agency pressure." Every screen should feel like advice from a knowledgeable friend — not a sales funnel and not a bureaucratic form.

1.3  Language & Tone
Address the seller as "você" (formal but warm — not the informal "tu")
Use plain language: "tratamos dos seus documentos" not "procedemos à obtenção da documentação necessária"
Positive framing: "Only 2 documents missing" not "Missing 2 documents"
Specificity over vagueness: "We requested your document on Monday; typical wait is 7–10 days" not "In progress"
English version: same principles, British spelling (seller base is UK/NL/DE)

2.2  DocCheck Tool (/verificar)
The DocCheck tool is the most important acquisition screen. It must be fast, simple, and end with a specific, personalised recommendation — not a generic one.
 
SCREEN 1: Introduction
  Headline: "Descubra que documentos precisa para vender"
  Sub: "8 perguntas rápidas. Resultado personalizado. Gratuito."
  CTA: "Começar"
  Progress bar: shown from this point (0/8)

SCREENS 2–9: Questions (one per screen)
  Layout: Question text (large) + 2–4 answer options (large touch targets)
  Progress bar advances with each answer
  Back button always visible
  No next button — answer selection auto-advances

SCREEN 10: Email capture (before showing result)
  Headline: "O seu relatório está pronto"
  Sub: "Introduza o seu email para ver os resultados"
  Input: Email field + CTA "Ver o meu relatório"
  Social proof: "Junte-se a X vendedores que já usaram o DocCheck"

SCREEN 11: Result page
  Section 1: Document checklist (green ticks for present, red flags for missing)
  Section 2: Recommended service with personalised price
    (if Q7=yes: show price with -€100 energy cert deduction)
  Section 3: Timeline estimate
  Section 4: Risk flags (if any)
  Primary CTA: "Encomendar o [recommended tier]"
  Secondary CTA: "Falar connosco primeiro" (WhatsApp link)

4. Operator Dashboard Screens
4.1  Operator Queue (/admin/queue)
PAGE HEADER
  "Painel de Operações" | Today's date
  Summary chips: [X Activas] [Y Bloqueadas] [Z Aguardam acção] [W Concluídas esta semana]

FILTERS BAR
  Status: All | Active | Blocked | Awaiting seller | Complete
  Tier: All | Standard | Premium | Express
  Search: by order ID, seller name, or address

ORDER TABLE (sortable columns)
  Columns:
    Order ID (link to detail)
    Seller name
    Property address / municipality
    Tier (badge: Standard / Premium / Express)
    Status (badge with colour: Active=blue, Blocked=red, Waiting=amber, Complete=green)
    Docs ready (e.g. "3/6")
    Days open
    Urgency flag (auto-calculated: > 15 days without completion = high)
  Each row: click to open order detail
  Rows update in real-time (Supabase Realtime subscription)

 
4.2  Order Detail (/admin/orders/[id])
LEFT PANEL: Property & Seller Info
  Property: artigo matricial, address, typology, area, VPT
  Seller: name, email, phone, language, residency, ownership type
  Order: tier, payment confirmed, Stripe reference, created date

CENTRE PANEL: Document Checklist
  Each document row:
    Document name
    Status dropdown: [Missing] [Requested] [Pending] [Complete] [Not Required]
    Date obtained / expected date
    Notes field (internal — not visible to seller)
    Upload button (for documents obtained by operator)
    Download button (for uploaded docs)
  "Mark order complete" button (active when all docs are Complete or Not Required)

RIGHT PANEL: Actions & Comms
  Quick actions:
    [Request seller action] → opens modal with action type + message
    [Flag as blocked] → opens modal with reason + estimated resolution date
    [Add internal note] → operator-only note, not visible to seller
    [View seller dashboard] → opens seller view (read-only)

  Comms log:
    All automated notifications sent (timestamp + channel + content)
    All manual messages sent by operator
    All seller responses (if via dashboard comms)
    Sorted newest first

  Supplier assignments:
    Perito: name, booking date, appointment date, cert received date
    Solicitador: name, brief, deliverable, status

 

5. Mobile UX Considerations
5.1  DocCheck on Mobile
Answer buttons must be minimum 48px height with 16px font — large enough for thumbs
One question per screen — do not show multiple questions on mobile
Progress indicator at top (e.g. "4 de 8 perguntas") — not a progress bar on narrow screens
Back button always accessible — top left, minimum 44×44px touch target
 
5.2  Document Upload on Mobile
Support camera capture directly (accept="image/*;capture=environment")
Show image preview before submission with "Parece bem?" confirmation
File size warning if > 10MB (Portal das Finanças PDFs are usually < 2MB)
Processing state: clear spinner + "A analisar o seu documento..." (30s max)
 
5.3  Dashboard on Mobile
Document tracker: cards stack vertically, full width, with large status badge
Download button: full-width on mobile, opens in new tab (PDF viewer)
SmartCMA price band: horizontal scroll chart or simplified vertical version on < 400px
All CTAs: minimum 48px height, full width on mobile
 
5.4  WhatsApp Deep Links
All WhatsApp CTAs: use wa.me/351XXXXXXXXX?text= format
Pre-fill the message with order reference: ?text=Olá, referência encomenda %23[ORDER_ID]
Test deep links on iOS and Android — behaviour differs between native and web
 

6. Error States & Edge Cases
6.1  Error States to Design
State
User-facing message
Action
OCR fails
"Não conseguimos ler o documento. Introduza os dados manualmente."
Show manual entry form
Payment fails
"O pagamento não foi processado. Por favor tente novamente."
Retry CTA + support link
Stripe webhook delayed
"A confirmar o pagamento... isto pode demorar um momento."
Polling + redirect on confirm
Document upload too large
"O ficheiro é demasiado grande (máx. 20MB). Comprima e tente novamente."
Compression tip shown
Session expired
"A sua sessão expirou. Por favor entre novamente."
Redirect to login, restore post-login
SmartCMA generation fails
"O relatório está a ser preparado. Receberá um email quando estiver pronto."
Queue for retry + email on complete

 
6.2  Empty States
New seller with no orders: show DocCheck CTA prominently — "Ainda não tem encomendas. Comece por verificar os seus documentos."
Operator queue with no active orders: "Sem encomendas activas. Bom trabalho!" with confetti or simple success state
 
6.3  Loading States
DocCheck submission: skeleton loader for result page (2–3 seconds for API call)
Dashboard: skeleton cards for document tracker (load order data async)
SmartCMA: progress indicator showing steps ("Analisando o mercado... Calculando comparáveis... Gerando relatório...")
